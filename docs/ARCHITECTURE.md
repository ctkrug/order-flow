# Architecture

A concise map of the codebase for anyone (including a future session) picking this
project back up. See `docs/VISION.md` for why it exists and `docs/DESIGN.md` for the
visual spec.

## Layout

```
engine/                 Rust matching engine, compiled to WASM
  src/lib.rs            simulate_market_order() + wasm-bindgen entry point + unit tests
  build-wasm.sh          cargo build (wasm32) + wasm-bindgen -> ../site/vendor/engine
  Cargo.toml             wasm-bindgen pinned to match the installed wasm-bindgen-cli

site/                   Vite + D3 static front end
  index.html             page structure (header, ladder panel, HUD, timeline strip)
  css/style.css          docs/DESIGN.md tokens + every component's visual states
  favicon.svg → public/favicon.svg   generated SVG monogram (see public/ below)
  data/ → public/data/   real historical L2 snapshot sequences (see data/README.md)
  vendor/engine/         gitignored; wasm-bindgen output regenerated by build-wasm.sh
  js/
    main.js               bootstraps the app; owns all mutable UI state and wiring
    engine.js              loadEngine(): memoized wasm init + simulateMarketOrder()
    book-view.js            createLadderView(): D3 data-join rendering of the ladder
    ladder-model.js          pure: buildLadderRows(), crossesWowThreshold()
    format.js                pure: formatUsd(), formatSize()
    timeline.js              pure: clampTimelineIndex(), formatSnapshotTime(), etc.
    tween.js                  pure: easeOutCubic(), tweenValue() (digit-roll counter)
    audio.js                 createAudio(): synthesized WebAudio SFX + mute persistence
    *.test.js                 vitest unit tests for every pure module above
  e2e/order-flow.spec.js      Playwright Firefox checks for live replay, failure recovery,
                              responsive bounds, and keyboard tab order
  playwright.config.js        starts Vite and defines the Firefox browser-test project

docs/                   VISION (why) / DESIGN (visual spec) / BACKLOG (stories) / this file
```

## Data flow (the wow moment, end to end)

1. `main.js` loads the wasm engine (`engine.js`) and both scenario JSON files
   (`public/data/snapshots-{calm,thin}.json`) in parallel on page load.
2. The user drags the size slider → an `input` event fires on every tick (no debounce).
3. `main.js` reads the current snapshot's book side (asks for buy, bids for sell) and
   calls `engine.simulateMarketOrder(levels, sideIsBuy, orderSize)`, which serializes the
   levels to JSON and calls into the WASM `simulate_market_order_js` export. The Rust side
   walks the book level-by-level (best price first) exactly like a real exchange matching
   engine and returns a `FillResult` (consumed levels, avg price, slippage cost, unfilled
   size).
4. `book-view.js`'s `createLadderView().render(...)` re-runs `buildLadderRows()`
   (`ladder-model.js`) to compute each row's taken/remaining state, then D3 data-joins
   that into the DOM: a two-segment depth bar (amber "taken" + side-colored "remaining"),
   flashing rows that just transitioned to consumed.
5. `main.js` updates the HUD (best/avg price, unfilled size) and tweens the slippage
   counter (`tween.js`) from its previously displayed value to the new one over ~120ms.
   `ladder-model.js`'s `crossesWowThreshold()` detects the exact zero-to-nonzero
   transition and triggers the one-time amber glow pulse + `audio.wowMoment()` chime.
6. `audio.js` also fires a pitch-stepping `levelTick()` per newly consumed level (via a
   callback threaded through `render()`) and a throttled `counterClick()` while slippage
   is nonzero.

Scenario switching and timeline scrubbing follow the same `update()` path. They
change which snapshot and side feed step 3, then rescale the order-size range to
the current book depth while preserving the slider's relative position.

## Why the wasm bindings aren't committed

`site/vendor/engine/` is gitignored. The site's `predev`/`prebuild` npm hooks
(`package.json`) shell out to `engine/build-wasm.sh`, which builds the `wasm32-unknown-unknown`
target and runs `wasm-bindgen` to regenerate it, so `npm run dev`/`build` always ships
a fresh build of the engine rather than the two ever drifting apart. CI's `site` job
installs the matching Rust target + `wasm-bindgen-cli` version before building for the
same reason. See `engine/Cargo.toml`'s `wasm-bindgen` pin and `engine/build-wasm.sh`'s
comments if this ever needs to move to a different toolchain.

## Testing

- `cargo test` (in `engine/`): the matching engine's fill algorithm, including the
  zero-slippage/multi-level/depth-exhausted boundaries.
- `npm test` (in `site/`): Vitest over every pure JS module (`format`, `ladder-model`,
  `timeline`, `tween`, and the mute/audio-synthesis logic in `audio`).
- `npm run test:coverage` (in `site/`): V8 coverage for those core pure modules;
  CI enforces an 85% line floor.
- `npm run test:e2e` (in `site/`): Playwright + Firefox starts the real Vite app and
  checks the WASM-to-D3 replay, missing-WASM recovery, 390/768/1440px widths, and
  keyboard focus order.

## Running locally

```sh
cd site
npm install
npm run dev     # predev regenerates the wasm bindings, then starts Vite
npm run build   # outputs a self-contained, subpath-relative dist/
npm test        # vitest
npm run test:coverage
npm run test:e2e
```
